INTRODUCTION

Standards are powerful means to realize interoperability among heterogeneous systems. The process of defining them is usually as follows. Interested stakeholders come together and decide to define a new standard for a given case. They build a consortium which drives this process and they create a standardization organization or they create a subgroup within an existing one. In periodic meetings, representatives from the different stakeholders come together, communicate their particular needs, and try to find a consensus. If the outcome, which can be e.g a protocol, a component specification, or a vocabulary1, is at a satisfying maturity level a specification document will be released. The adopters will then create implementations of this standard. Possibly, they need to adapt their already existing systems.

Apparently, the overall process is quite cumbersome and long lasting. The more stakeholders are involved and the more existing proprietary products are affected the more is this process exacerbated.

The open and dynamic World Wide Web, on the contrary, demonstrates that with a minimalistic standard set and flexible de facto standards interoperability is also possible to some extend. This is mainly enabled by focused applications and well documented specification pages. In some cases these de facto standards become real standards. However, the main idea is that they are not created in a top-down approach as in traditional standardization activities. Concretely, the implementation is not based on a predefined standard, but the standard is based on the adoption and the experience with existing implementations. That makes them more practical and avoid overly engineered standards like CORBA, CAMEL (Customised Applications for Mobile networks Enhanced Logic), etc. We consider this as a bottom-up approach for defining a standard.

It is worth mentioning that even within the Web context the danger of overly engineered standards is also given. The vision of the Semantic Web for example, caused the enthusiastic creation of standards like the Web Ontology Language and the Rule Interchange Format to represent knowledge and rules. It remains questionable if and when this standards will be really broadly adopted and if they are really practical enough to be used in various information systems. In contrast, positive examples likes Schema.org2 clearly demonstrate that a practice oriented approach is very effective. The definition, implementation and the usage is integrated pragmatically and not organized sequentially. Additionally, the open and participatory nature of the Web is leveraged by using the GitHub collaboration platform for managing feedback and contributions, versioning, publishing releases and tracking issues.

Inspired by the success of Schema.org we advocate to revolutionize approaching standardization to enable interoperability. In this regard, we propose a set of practices to create a common vocabulary to bridge heterogeneity among systems. Furthermore, the endeavor to derive this methodology led to valuable insights as a by-product. Therefore, the applied bottom-up and pragmatically best-practice oriented technique is presented in detail.

The remainder of this paper is organized as follows. In sec:bottom-up we present our approach to derive a methodology from best practices. In sec:vocab-analysis we present an analysis of the most widely vocabularies and we extract some useful data to be considered later for our approach. In sec:best-practices we propose our set of best-practices for collaborative vocabulary engineering. A review of the state of the art is presented in sec:related-work. We conclude this paper in sec:conclusion with shedding light on the critical aspects of this paper and provide an outlook to meaningful extensions of this work.
